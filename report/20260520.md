## go-yaml

- [ ] [#345 Stack overflow when custom UnmarshalYAML wraps unmarshal callback error](https://github.com/yaml/go-yaml/issues/345)
  A crash-level bug: custom `UnmarshalYAML` implementations that wrap and re-return the error from the unmarshal callback trigger infinite recursion and blow the stack. Affects any library doing `fmt.Errorf("...: %w", unmarshal(...))`. Still unresolved with no fix merged.
  (updated: 2026-05-08T14:54:24Z, appearance #2)

- [ ] [#342 BC break: yaml construct error on empty/comment-only stream](https://github.com/yaml/go-yaml/issues/342)
  Parsing empty or comment-only YAML streams now raises an error in v4, breaking every consumer that previously treated no-content as a valid (nil) load. Unresolved and open.
  (updated: 2026-05-05T15:58:34Z, appearance #2)

- [ ] [#337 Folded block scalars gain one newline per yaml.Node round-trip](https://github.com/yaml/go-yaml/issues/337)
  Each load-and-save cycle through `yaml.Node` appends an extra newline to folded block scalars, corrupting data over repeated passes. Disqualifies go-yaml from use in format-preserving editing tools. Still open.
  (updated: 2026-04-21T20:49:31Z, appearance #2)

- [ ] [PR #332 Fix: KnownFields not being respected inside custom UnmarshalYAML()](https://github.com/yaml/go-yaml/pull/332)
  Open fix for a correctness bug where the `KnownFields` strict-mode option is silently ignored inside custom `UnmarshalYAML` implementations, potentially masking unknown-field errors in deeply nested decoding paths. Has been open and unmerged for several weeks.
  (updated: 2026-05-12T12:33:53Z, appearance #2)

## yamlscript

- [ ] [#267 macOS arm64: installer Makefile uses cp libys*.so* (Linux glob) — fails on macOS](https://github.com/yaml/yamlscript/issues/267)
  The installer's Makefile hardcodes a Linux-specific glob (`libys*.so*`) that fails on macOS where libraries use `.dylib` extensions, silently breaking installs on Apple Silicon. A one-line fix is proposed in the issue but remains unmerged.
  (updated: 2026-05-07T16:54:08Z, appearance #2)

- [ ] [#266 Blog post "YAML best practices": whitespaces after `:` and before `#` are not rendered](https://github.com/yaml/yamlscript/issues/266)
  The yamlscript.org blog post on YAML best practices drops significant whitespace around `:` and `#` in rendered code examples, making it look like those characters are disallowed — potentially misleading to new users learning YAML conventions. Still open.
