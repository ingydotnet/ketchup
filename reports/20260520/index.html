    go-yaml¶

    • #345 Stack overflow when custom UnmarshalYAML wraps unmarshal callback error A crash-level bug: custom UnmarshalYAML implementations that wrap and re-return the error from the unmarshal callback trigger infinite recursion and blow the stack. Affects any library doing fmt.Errorf("...: %w", unmarshal(...)). Still unresolved with no fix merged. (updated: 2026-05-08T14:54:24Z, appearance #2)

    • #342 BC break: yaml construct error on empty/comment-only stream Parsing empty or comment-only YAML streams now raises an error in v4, breaking every consumer that previously treated no-content as a valid (nil) load. Unresolved and open. (updated: 2026-05-05T15:58:34Z, appearance #2)

    • #337 Folded block scalars gain one newline per yaml.Node round-trip Each load-and-save cycle through yaml.Node appends an extra newline to folded block scalars, corrupting data over repeated passes. Disqualifies go-yaml from use in format-preserving editing tools. Still open. (updated: 2026-04-21T20:49:31Z, appearance #2)

    • PR #332 Fix: KnownFields not being respected inside custom UnmarshalYAML() Open fix for a correctness bug where the KnownFields strict-mode option is silently ignored inside custom UnmarshalYAML implementations, potentially masking unknown-field errors in deeply nested decoding paths. Has been open and unmerged for several weeks. (updated: 2026-05-12T12:33:53Z, appearance #2)

    libyaml¶

    • #325 compile error C4703: potentially uninitialized local pointer variable 'input' used MSVC compile error with a companion fix in PR #326 that also remains unmerged, blocking clean Windows builds. Especially relevant now that 0.2.6-rc.1 is out — packagers on the MSVC toolchain need this resolved before stable. (updated: 2026-03-23T22:41:37Z, appearance #2)

    • PR #327 Add libFuzzer harnesses for parser, loader, scanner and emitter APIs Still-open PR adding broad fuzzing coverage across the public API surface, relevant in the run-up to the 0.2.6 release. Fuzzer harnesses are the kind of infrastructure that surfaces safety issues before a stable cut. Awaiting maintainer review. (updated: 2026-03-24T06:39:02Z, appearance #2)

    • #324 libyaml_dumper_fuzzer: parser leak on emitter init failure Fuzzer-discovered memory leak: a parser resource is not freed when emitter initialization fails. Still unresolved and stale since March. (updated: 2026-03-08T21:48:47Z, appearance #2)

    yamlscript¶

    • #267 macOS arm64: installer Makefile uses cp libys.so (Linux glob) — fails on macOS The installer's Makefile hardcodes a Linux-specific glob (libys*.so*) that fails on macOS where libraries use .dylib extensions, silently breaking installs on Apple Silicon. A one-line fix is proposed in the issue but remains unmerged. (updated: 2026-05-07T16:54:08Z, appearance #2)

    • #266 Blog post "YAML best practices": whitespaces after : and before # are not rendered The yamlscript.org blog post on YAML best practices drops significant whitespace around : and # in rendered code examples, making it look like those characters are disallowed — potentially misleading to new users learning YAML conventions. Still open. (updated: 2026-04-17T12:11:46Z, appearance #2)

    exercism/yamlscript¶

    • #53 Two-Fer exercise validation failing to run A compile error in the two-fer exercise test file causes validation to fail with no subtests run, blocking learners on a foundational Exercism exercise. Open since March with no fix merged. (updated: 2026-03-05T00:08:55Z, appearance #2)

    • PR #54 bin/verify-exercises-in-docker Open PR adding Docker-based exercise verification that also tidies the two-fer test file — the direct fix path for issue #53. Still awaiting review. (updated: 2026-05-08T11:42:04Z, appearance #2)

    • #1 Adopt official yaml test suite Request to run the full official yaml-test-suite to improve YAML 1.2 spec compliance confidence. Still open with no PR linked. (updated: 2026-03-03T18:08:46Z, appearance #2)
